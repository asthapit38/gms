app/

Routes and page components only.

Pages should compose components and call hooks/server actions. They should not contain large UI blocks or complex business logic inline.

components/ui/

Auto-generated by shadcn/ui CLI. Do not hand-edit files here. Add new components via npx shadcn@latest add <component>.

components/shared/

Create a shared component only when:


It is reused in more than one route or feature
It represents a clear UI concept like MemberCard, AttendanceTable, StatCard, or PageHeader
Extracting it makes a page significantly easier to read


Do not create tiny one-off wrapper components prematurely. When unsure, ask:


"Should this UI be extracted into a reusable component, or should I keep it inside the current page for now?"



db/schema/

One schema file per domain. Example:

db/schema/

  attendance.ts
  subscriptions.ts

Export all schemas from db/schema/index.ts so Drizzle migrations stay predictable.

Database Rules (Drizzle ORM + PostgreSQL)


Define all schemas in db/schema/ — one file per domain
Export everything from db/schema/index.ts
Create a single Drizzle client instance in db/index.ts and import from there
Use Drizzle migrations (drizzle-kit generate → drizzle-kit migrate) — do not use db.push() in production
Keep schema types simple and explicit — avoid over-normalizing early
Use drizzle-zod or manual Zod schemas for input validation where needed


Schema Example Pattern

ts// db/schema/members.ts
import { pgTable, uuid, text, timestamp, pgEnum } from "drizzle-orm/pg-core";

export const memberStatusEnum = pgEnum("member_status", ["active", "inactive", "suspended"]);

export const members = pgTable("members", {
  id: uuid("id").primaryKey().defaultRandom(),
  name: text("name").notNull(),
  email: text("email").notNull().unique(),
  phone: text("phone"),
  status: memberStatusEnum("status").default("active").notNull(),
  createdAt: timestamp("created_at").defaultNow().notNull(),
  updatedAt: timestamp("updated_at").defaultNow().notNull(),
});


Authentication Rules (better-auth)


Use better-auth for all authentication — do not build custom auth
Server config lives in lib/auth.ts
Client config lives in lib/auth-client.ts
Protect admin routes using middleware or layout-level session checks
Two roles: admin and member — enforce role checks server-side, not just client-side
Never expose secrets in client components


Session Access Pattern

ts// Server Component or Server Action
import { auth } from "@/lib/auth";
import { headers } from "next/headers";

const session = await auth.api.getSession({ headers: await headers() });
if (!session) redirect("/login");

TanStack Query Setup


Initialize the client in lib/query-client.ts
Wrap the app in QueryClientProvider inside a client boundary
Use descriptive, consistent query keys:


ts// Good
["members", "list"]
["members", memberId]
["attendance", { date: "2025-06-01" }]

Server Action + Mutation Pattern

ts// Server Action
"use server";
export async function createMember(data: NewMember) {
  await db.insert(members).values(data);
  revalidatePath("/admin/members");
}

// Client component
const { mutate } = useMutation({
  mutationFn: createMember,
  onSuccess: () => queryClient.invalidateQueries({ queryKey: ["members"] }),
});

TypeScript Rules


Use TypeScript strictly — enable strict: true in tsconfig.json
Avoid any — use unknown and narrow types properly
Keep types simple and readable — avoid deeply nested generics early on
Define shared types in types/index.ts
Infer Drizzle table types using typeof table.$inferSelect and typeof table.$inferInsert


ts// types/index.ts
import { members } from "@/db/schema";

export type Member = typeof members.$inferSelect;
export type NewMember = typeof members.$inferInsert;
